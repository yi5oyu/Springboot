# HTTP/HTTPS

`웹 브라우저와 웹 서버가 HTML, 이미지, JSON 데이터 등을 주고받을 때 사용`

## HTTP(HyperText Transfer Protocol)

`:80`

주고받는 데이터가 암호화되지 않은 평문(Plain Text) 형태이기 때문에 중간에 데이터를 가로채면 비밀번호 같은 정보가 그대로 노출될 위험 있음

## HTTPS(HyperText Transfer Protocol Secure)

`:443`

HTTP에 데이터 암호화(SSL/TLS) 기능을 추가하여 보안을 대폭 강화한 프로토콜

### [SSL/TLS](https://github.com/yi5oyu/Study/blob/main/WS/Nginx/HTTPS/SSL)

- 암호화(Encryption): 중간에서 데이터를 가로채더라도(스니핑) 암호화되어 있어 그 내용(비밀번호, 개인정보 등)을 읽을 수 없게 만듬
- 인증(Authentication): 클라이언트(브라우저)가 접속한 서버가 가짜 피싱 사이트가 아니라 신뢰할 수 있는 서버임을 보증
- 무결성(Integrity): 전송되는 데이터가 중간에 누군가에 의해 위조되거나 변조되지 않았음을 확인

```
통신 과정
1. Client Hello(클라이언트 요청)
 - 클라이언트 랜덤 데이터: 비밀 키를 만들 때 사용할 난수 하나 생성
 - Cipher Suite(암호화 알고리즘 모음): 지원하는 암호화 방식들의 리스트를 보냄
 - 세션 ID: 이전에 연결했던 적이 있다면 빠르게 연결할 수 있음
2. Server Hello(서버 응답)
 - 서버 랜덤 데이터: 사용할 난수 하나 생성
 - 선택된 암호화 알고리즘: 암호화 방식 선택
3. Server Certificate(서버 인증서 전달)
 - SSL 인증서(공개키)를 클라이언트에게 보냄
4. Server Hello Done(완료)
5. Client Key Exchange(비밀키 생성/전달)
 - Pre-Master Secret(예비 마스터 키) 생성: 주고받은 클라이언트 랜덤 데이터/서버 랜덤 데이터 조합헤 임시 비밀 키를 만듬
 - 서버의 공개키로 예비 마스터 키를 암호화해서 서버로 전송(서버의 개인키로만 풀 수 있음)
6. Session Key(세션 키)
 - 서버는 개인키로 클라이언트가 보낸 암호를 품
 - 통신에 사용할 세션키를 클라이언트/서버가 각자 똑같이 생성
7. Secure Connection(통신 시작)
```

# WebSocket

`:80/:443 실시간 양방향 통신`

웹소켓은 한 번 연결되면 서버와 클라이언트가 자유롭게 양방향으로 데이터를 주고받을 수 있음

```http
연결

- 클라이언트 요청(HTTP Request): 서버로 HTTP GET 요청 보냄

GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket                     <-- 웹소켓으로 업그레이드
Connection: Upgrade                    <-- 통신 방식 업그레이드
Sec-WebSocket-Key: dGhlIHNhbXb25jZQ==  <-- 보안을 위한 랜덤 키
Sec-WebSocket-Version: 13

- 서버 응답(HTTP Response): 통신 규약을 바꾸겠다는 응답

HTTP/1.1 101 Switching Protocols       <-- 101번 상태 코드
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMB25K+xOo=  <-- 클라이언트가 준 키를 조합해 만든 암호

> HTTP 연결은 종료되고 웹소켓(ws:// 또는 wss://) 연결
```
