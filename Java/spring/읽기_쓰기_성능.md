## 읽기 전략

### 클라이언트

`사용자와 가장 가까운 계층에서 요청을 처리해 서버로 오는 트래픽 자체를 줄이는 전략`

- 브라우저 캐시: 정적 리소스(HTML, CSS, JS, 이미지)를 클라이언트(브라우저)에 저장하여 서버 요청을 제거
- CDN(Content Delivery Network): 엣지 서버(Edge Server)에서 콘텐츠를 서빙

### 어플리케이션

`DB 부하를 줄이기 위해 메모리 영역 사용`

- 로컬 캐시: 서버 내부 메모리(Heap)에 데이터를 저장
- 글로벌 캐시: 여러 서버가 공유하는 외부 저장소(Redis)를 사용

### 데이터베이스

`Disk I/O 병목을 해결하기 위한 분산 처리`

- 복제(Replication): DB를 Master(쓰기)와 Slave(읽기)로 분리해 트래픽 분산
- 인덱싱(Indexing): 쿼리에 필요한 모든 컬럼을 인덱스가 포함하고 있어 실제 데이터 블록 접근 없이 인덱스 스캔만으로 조회 완료

## 쓰기 전략

### 클라이언트

`서버로 전송되는 쓰기 요청 횟수 자체를 줄이거나 체감 속도를 높이는 전략`

- 디바운싱(Debounce): 연이어 호출되는 함수들 중 마지막만 실행
- 쓰로틀링(Throttle): 일정 시간 동안 이벤트를 한 번만 실행
- 낙관적 UI(Optimistic UI): 서버의 성공 응답을 기다리지 않고 UI를 먼저 업데이트해 사용자에게 즉각적인 반응 제공
 
### 어플리케이션

`DB 병목을 막기 위해 중간에서 버퍼링하거나 비동기로 처리`

- 비동기 처리: 사용자에게 응답을 먼저 주고 실제 쓰기 작업은 백그라운드에서 처리
- 쓰기 지연: 데이터를 캐시(Redis)에 먼저 빠르게 저장하고 일정 주기마다 DB에 한꺼번에 반영
- 메시지 큐 버퍼링: Kafka와 같은 큐에 요청을 쌓아두고 서버가 처리 가능한 속도로 가져와서 처리

### 데이터베이스

`대량의 데이터를 효율적으로 저장하고 분산하는 기술`

- 벌크/배치 연산: 여러 개의 데이터를 묶어서 한 번의 쿼리로 저장
- 샤딩(Sharding): 데이터를 여러 물리적 서버에 쪼개서 저장(단일 테이블의 용량이 너무 커지면 쓰기 속도가 저하됨)
- 파티셔닝(Partitioning): 하나의 큰 테이블을 논리적으로 작은 단위로 쪼개서 관리
