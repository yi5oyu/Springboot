# 비동기 (Asynchronous)

`하나의 서비스에서 기능을 순차 처리해 응답 속도 느려지기 때문에 작업이 완료될 때까지 기다리지 않고 다음 코드를 실행하는 방식` 

</div>

## WebMVC (@Async + CompletableFuture)

```text
// CompletableFuture 반환

1. 클라이언트 HTTP 요청

2. WAS(톰캣) HTTP 요청 수신
 - 스레드 풀에서 요청 처리할 스레드 할당

3. Controller, Service 로직 실행

4. @Async 메서드 호출
 - 프록시 객체에서 비동기 스레드 풀(ThreadPoolTaskExecutor)에 작업 넘긴 후 톰캣 스레드는 풀에 반환
 - 비동기 스레드는 @Async 메서드 로직 실행

5. TaskExecutor 비동기 스레드 작업 완료
 - 비동기 스레드가 @Async 로직 실행을 완료하고 결과를 반환

6. 클라이언트에 HTTP 응답
 - 작업 완료 이벤트가 발생하면 톰캣 스레드 풀에서 새로운 스레드를 다시 할당해 결과 응답
```

### @Async

`AOP(Aspect-Oriented Programming) 기반의 프록시(Proxy) 패턴` 

`@EnableAsync 비동기 기능 활성화`

```java
// 둘 중 한 방법 사용
// 1. main 클래스
@EnableAsync 
@SpringBootApplication
public class MyApplication {
    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
}

// 2. 설정 클래스
@EnableAsync    
@Configuration  
public class AsyncConfig {

}
```

`@Async 사용`

```java
@Service
public class ExService {
    @Async
    public void ex() {
        // 로직
    }
}
```

`@Async 프록시 객체`

```java
// Spring CGLIB가 ExService를 상속받아 만든 프록시 클래스
public class ExService$$EnhancerBySpringCGLIB extends ExService {
    // 진짜 객체(Target)
    private final ExService target; 
    
    // 비동기 스레드 풀 주입
    private final AsyncTaskExecutor executor; 

    public ExService$$EnhancerBySpringCGLIB(ExService target, AsyncTaskExecutor executor) {
        this.target = target;
        this.executor = executor;
    }

    // 원본 메서드를 오버라이딩
    @Override
    public void ex() {
        // 프록시가 가로채서 수행하는 추가 로직

        // 진짜 실행해야 할 원본 메서드 Runnable(or Callable)로 생성 
        Runnable task = new Runnable() {
            @Override
            public void run() {
                try {
                    // 진짜 객체 메서드 호출
                    target.ex(); 
                } catch (Throwable ex) {
                    // 예외 발생 시 AsyncUncaughtExceptionHandler
                    handleException(ex); 
                }
            }
        };

        // Executor 스레드 풀 (비동기 처리)
        executor.submit(task);

        // 메인 톰캣(WAS) 스레드 리턴되어 클라이언트 응답
        return; 
    }
}

```

### SimpleAsyncTaskExecutor

`요청이 올 때마다 자동으로 매번 새로운 스레드 생성/소멸됨`

- 비동기 요청에 따라 새 스레드가 계속 생겨 OutOfMemoryError 발생할 수 있음

### ThreadPoolTaskExecutor

`비동기를 위해 애플리케이션 레벨에서 사용하는 스레드 풀`

```java
public interface AsyncConfigurer {
	@Nullable
	default Executor getAsyncExecutor() { return null; }

	@Nullable
	default AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() { return null; }
}
```

`ThreadPoolTaskExecutor 설정`

```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(20);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("async-");

        // TaskDecorator 설정(SecurityContext 전파)
        executor.setTaskDecorator(new SecurityContextTaskDecorator());

        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());

        executor.initialize();
        return executor;
    }

    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (throwable, method, params) -> {
            System.err.println("비동기 메서드 예외 발생: " + method.getName());
            throwable.printStackTrace();
        };
    }
}
```

`비동기 스레드 처리 과정`

```text
// Core 10, Queue 100, Max 20

1. 요청 (1 ~ 10)
 - 미리 만들어둔 10개 코어 스레드가 처리

2. 트래픽 증가로 요청 증가 (11 ~ 110)
 - 코어 스레드가 모두 일하는 중이라면 나머지 요청 100개 큐에 쌓여 대기

3. 트래픽 증가로 요청 증가 (110 ~ 120)
 - 큐가 가득 차 임시 스레드를 Max인 20까지 추가 생성

4. 요청 초과
 - 모든 스레드(20)와 큐(100)가 사용중이면 새로운 요청은 예외(RejectedExecutionException) 발생
```

`4가지 거절 정책`

```
1. AbortPolicy (스프링 기본값)
 - 새로운 작업 거절하고 RejectedExecutionException 예외 발생
 - 예외 처리(try-catch or .exceptionally())하지 않으면 사용자에게 서버 에러(500)가 발생하거나 스레드 멈춤

2. CallerRunsPolicy
 - 백프레셔(Backpressure): 비동기 작업을 호출한 스레드(톰캣 스레드)에게 작업 다시 넘김
 - 비동기 + 논블로킹으로 동작하려는 의도와 달리 동기 + 블로킹으로 동작하게 됨

3. DiscardPolicy
 - 예외도 발생시키지 않고 큐에 들어가지 못한 새로운 작업을 그냥 버림

4. DiscardOldestPolicy
 - 큐에서 가장 오래 기다리고 있던 작업(맨 앞)을 하나 버리고 새로운 작업을 큐에 넣음
```

### CompletableFuture

`비동기 작업 시작`

```java
// 생성한 커스텀 스레드 풀 주입
private final Executor getAsyncExecutor;

// 반환값 없는 경우: CompletableFuture.runAsync()

CompletableFuture<Void> future = CompletableFuture.runAsync(() -> {
    System.out.println("비동기 작업 실행 중... ");
}, getAsyncExecutor);

// 반환값 있는 경우: CompletableFuture.supplyAsync()

CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    // 로직
    return "응답";
}, getAsyncExecutor);
```

`체이닝 패턴`

```java
// .thenApply() (결과를 받아 반환) 

CompletableFuture.supplyAsync(() -> "Hello")
    .thenApply(String::length);

// .thenAccept() (결과를 받아 처리(반환 x)) 

CompletableFuture.supplyAsync(() -> "Task Result")
    .thenAccept(result -> {
        // DB 저장, 출력 등...
    });
```

`병렬 처리`

```java
// .thenCompose() (여러 개 비동기 작업 순차적 실행) 

// 유저 정보를 가져온뒤 그 정보로 번호 조회 비동기로 실행
CompletableFuture<Number> getNumberFuture = getUserInfo()
    .thenCompose(user -> getNumberAsync(user.getId()));

// .thenCombine() (독립적인 작업을 병렬로 실행 후 합칠 때) 

CompletableFuture<String> userFuture = getUserAsync();
CompletableFuture<String> orderFuture = getOrderAsync();

// 서로 다른 두 작업이 병렬로 실행되고 모두 끝났을 때 콜백 실행
CompletableFuture<String> combined = userFuture.thenCombine(orderFuture, (user, order) -> {
    return user + "님의 주문: " + order;
});

// .allOf() (여러 개의 작업 모두 완료 후 처리)

CompletableFuture<String> task1 = callApiA();
CompletableFuture<String> task2 = callApiB();
CompletableFuture<String> task3 = callApiC();

CompletableFuture<Void> allFutures = CompletableFuture.allOf(task1, task2, task3);

// 여러 개의 외부 API를 동시에 호출하고 모든 응답이 도착했을 때 처리
allFutures.thenAccept(v -> {
    // 실행
    return "응답";
});
```

`예외 처리`

```java
CompletableFuture.supplyAsync(() -> {
    if (true) throw new RuntimeException("API 호출 실패");
    return "응답";
}).exceptionally(ex -> {
    System.err.println("에러 발생: " + ex.getMessage());
    return "실패 시 응답"; 
});
```
